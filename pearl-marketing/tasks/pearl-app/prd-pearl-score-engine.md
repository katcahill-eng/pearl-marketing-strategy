# PRD: Pearl Score Calculation Engine

## Introduction/Overview

The Pearl Score is the central metric of the app â€” a 0-1000 score rating a home's performance across five SCORE pillars (Safety, Comfort, Operations, Resilience, Energy). Currently the score is hardcoded at 625 in MyHomeScreen. This feature replaces that with a dynamic calculation engine that derives the score from factors generated by FactorEngine, persists score history for animation, and updates in real time as the user adds equipment, connects devices, logs symptoms, or completes onboarding.

## Goals

- Calculate a dynamic 0-1000 Pearl Score from the FactorEngine's helping/hurting factors
- Each SCORE pillar contributes up to 200 points (5 Ă— 200 = 1000)
- Score updates reactively when underlying data changes (equipment, readings, symptoms, onboarding)
- Persist previous score in AsyncStorage so ScoreDisplay can animate score changes
- Wire the calculated score into MyHomeScreen replacing the hardcoded 625
- Provide per-pillar sub-scores visible in the PillarDetailScreen

## User Stories

### US-115: Create ScoreEngine service
**Description:** As a developer, I need a pure-function service that takes Factor[] and returns a total score (0-1000) plus per-pillar sub-scores (0-200 each).

**Acceptance Criteria:**
- [ ] Create `src/services/ScoreEngine.ts` with `calculateScore(factors: Factor[]): ScoreResult`
- [ ] `ScoreResult` type: `{ total: number; pillars: Record<Pillar, number> }`
- [ ] Each pillar starts at a base of 125 points (625 total baseline)
- [ ] Each helping factor adds points: info +10, attention +15, action +20
- [ ] Each hurting factor subtracts points: info -5, attention -15, action -30
- [ ] Per-pillar scores clamped to 0-200 range
- [ ] Total score is sum of all 5 pillar scores (0-1000)
- [ ] Export `calculateScore` and `ScoreResult` from `src/services/index.ts`
- [ ] Typecheck passes

### US-116: Persist score history in AsyncStorage
**Description:** As a user, I want to see my score animate from the previous value to the new value so I can see the impact of my actions.

**Acceptance Criteria:**
- [ ] Create `src/context/ScoreContext.tsx` providing `{ score, previousScore, pillarScores, refreshScore }`
- [ ] Context calls `generateAllFactors` + `calculateScore` using data from existing contexts
- [ ] Stores current score under `pearl_score_current` and previous under `pearl_score_previous` in AsyncStorage
- [ ] On first load, previous score is null (no animation)
- [ ] When score changes, old current becomes new previous, new score becomes current
- [ ] Export from `src/context/index.ts`
- [ ] Add `ScoreProvider` to provider chain in `App.tsx` (after existing providers it depends on)
- [ ] Typecheck passes

### US-117: Wire dynamic score into MyHomeScreen
**Description:** As a user, I want to see my real Pearl Score on the My Home tab so I understand my home's actual performance.

**Acceptance Criteria:**
- [ ] Remove hardcoded `const homeScore = 625` from MyHomeScreen
- [ ] Import and use `useScore` from ScoreContext
- [ ] Pass `score` and `previousScore` to `ScoreDisplay` component
- [ ] Score updates when equipment, symptoms, readings, or onboarding data change
- [ ] Typecheck passes

### US-118: Show per-pillar sub-scores in PillarDetailScreen
**Description:** As a user, I want to see each pillar's individual score so I understand which areas need the most improvement.

**Acceptance Criteria:**
- [ ] Import `useScore` in PillarDetailScreen
- [ ] Display pillar sub-score (0-200) next to the status badge in the header
- [ ] Show "X / 200" format with the pillar's status color
- [ ] Typecheck passes

## Functional Requirements

- FR-1: ScoreEngine is a pure function â€” no side effects, no async, deterministic output for given input
- FR-2: Base score per pillar is 125 (producing 625 total with no factors â€” matches current mock)
- FR-3: Factor points scale with severity: helping factors add, hurting factors subtract
- FR-4: Per-pillar scores never go below 0 or above 200
- FR-5: Total score is the sum of all 5 pillar scores
- FR-6: ScoreContext recalculates whenever underlying contexts update
- FR-7: Score persists in AsyncStorage with `pearl_` prefix convention
- FR-8: Previous score enables the existing ScoreDisplay animation infrastructure

## Non-Goals

- No server-side score calculation or validation
- No score sharing or export functionality
- No certification tier logic (separate future feature)
- No score breakdown tooltip beyond per-pillar sub-scores

## Technical Considerations

- FactorEngine already generates all the factor data needed â€” ScoreEngine just needs to consume Factor[]
- ScoreContext depends on DeviceContext, EquipmentContext, SymptomContext, and OnboardingContext
- Must be placed after those providers in the App.tsx chain
- ScoreDisplay component already has full animation support â€” just needs real data
- Use `useMemo` for score calculation to avoid recalculating on every render
- Follow existing patterns: AsyncStorage with `pearl_` prefix, barrel exports, useTheme()
