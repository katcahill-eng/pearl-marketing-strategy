## Codebase Patterns
- MarcomsBot lives in `pearl-bot/` directory with its own package.json
- Intake flow handler in `src/handlers/intake.ts` — central dispatch for all conversation messages (gathering, confirming, complete, cancelled)
- DM handler checks for active conversation BEFORE intent detection — ongoing conversations route directly to intake
- `event.user` in Bolt.js `app_mention` events can be `undefined` — always use `?? ''` fallback
- Uses Bolt.js socket mode for Slack connectivity
- TypeScript strict mode enabled, target ES2022
- Run `npm run typecheck` (tsc --noEmit) to verify types
- Run `npm run build` (tsc) to compile to dist/
- Run `npm run dev` (tsx watch) for local development
- Database: better-sqlite3 with WAL mode, stored at `pearl-bot/data/marcomsbot.db`
- Export `Database` type from better-sqlite3 as `DatabaseType` to avoid TS4023 export naming errors
- Prepared statements are pre-compiled in a `stmts` object for performance
- All helper functions are named exports; `db` instance is the default export
- Event handlers go in `pearl-bot/src/handlers/` — one file per event type, registered in index.ts
- Intent detection uses keyword regex patterns in `handlers/intent.ts`
- All bot replies must use `thread_ts` (either `event.thread_ts` or `event.ts`) to stay threaded
- DM handler filters on `event.channel_type === 'im'` and skips `event.subtype` for edits/bots
- ConversationManager is the central state machine class — load/create via static `load()` or `new`, persist with `.save()`
- Required intake fields in order: division, request_type, description, deliverables, due_date, urgency
- `collected_data` is JSON-stringified CollectedData; array fields (deliverables, assets_mentioned) need Array.isArray checks
- Claude integration in `src/lib/claude.ts`: `interpretMessage()` for NLP extraction, `classifyRequest()` for quick/full classification
- Anthropic SDK auto-reads ANTHROPIC_API_KEY from env — no explicit config needed
- Claude model for real-time Slack: claude-sonnet-4-20250514 (fast, cost-effective)
- Brief generation in `src/lib/brief-generator.ts`: `generateBrief(collectedData, classification, requesterName?)` → markdown string
- Brief templates use system prompts with exact markdown structure; Claude fills narrative sections from collected data
- Monday.com integration in `src/lib/monday.ts`: GraphQL API at `https://api.monday.com/v2`, exports `createQuickRequestItem()`, `createFullProjectGroup()`, `searchItems()`
- Monday.com `column_values` must be JSON-stringified when passed as mutation variable
- No extra npm deps for Monday — uses native `fetch`
- Post-confirmation workflow in `src/lib/workflow.ts`: `executeWorkflow()` orchestrates brief → Drive → Monday → DB; `buildCompletionMessage()` formats Slack output
- Both conversation and form intake paths should use `executeWorkflow()` for the downstream pipeline
- Bolt.js event handlers provide `client` (WebClient) alongside `say` — use `client.chat.postMessage` for posting to arbitrary channels
- Channel notifications are built by `src/lib/notifications.ts` (pure formatting) — Slack posting is done by the handler
- Status/search handlers in `src/handlers/status.ts` and `src/handlers/search.ts` — search local DB + Monday.com in parallel, merge results by name
- Search handler focuses on links (brief, folder, Monday URL); status handler focuses on progress (status, assignee, due date)
- Schema evolution: use `ALTER TABLE ADD COLUMN` wrapped in try/catch for adding columns without migration framework
- Timeout handler in `src/handlers/timeout.ts`: `checkTimeouts(client)` runs on 15-min interval via `setInterval` in index.ts
- `timeout_notified` column (0/1) on conversations table tracks two-phase timeout: notify → auto-cancel
- Duplicate thread detection: `pendingDuplicateChecks` in-memory Map in intake.ts tracks prompts; both messages.ts and mentions.ts check `hasPendingDuplicateCheck()` before intent detection
- Slack deep links: `https://slack.com/archives/{channelId}/p{tsWithoutDot}` — remove dots from thread_ts
- Webhook server in `src/lib/webhook.ts`: standalone HTTP server on WEBHOOK_PORT (default 3100), POST /webhook/intake for form submissions — uses same `executeWorkflow()` as conversation path
- Error handling: all Slack event handlers (mentions.ts, messages.ts) have top-level try/catch; intake.ts uses inner function pattern for error isolation; all user-facing errors include form link fallback (INTAKE_FORM_URL) and #marketing-team alternative
- Never expose raw error messages to users — all external-facing error responses are sanitized; console.error with `[module]` prefix for debugging
- Configuration: `src/lib/config.ts` centralizes all env var access — import `config` object, never use `process.env` directly in other modules
- Config validates required env vars on startup (exits with clear error) — import config early in index.ts (right after dotenv)
